# Proposal: `dispatch` — make git the sole trigger-and-report channel

## The request (verbatim)

> 에이전트 트리거와 보고를 위한 대화 채널을 git에 전적으로 의존하게 하려는거야
> 가능하게 설계해

("I want the conversation channel for agent triggering and reporting to depend
entirely on git. Design it so it's possible.")

## Constraints that shape the build

These are settled in the conversation that produced this proposal; they are the
decisions, not open questions.

1. **`report == trigger`.** Outbound is a git write and inbound is a git event,
   so an agent's report *is* the next actor's trigger. This identity only holds
   if git is the *sole* channel — a single ephemeral chat message for a handoff
   breaks it, because no agent can subscribe to it. "Entirely git," not "mostly."
2. **The oracle stays above and is a git act.** Acceptance is `merge` / issue
   `close`; it is enforced by git permissions, not app logic. An agent never
   accepts its own work. This is the position paper's layer separation (§4)
   realized at the transport level — a human's feedback-generation and an
   agent's generation are the same kind of event on the same bus.
3. **No self-verification, mechanically.** The generating layer does not check
   itself (§3). In transport terms: an agent may not resolve, merge, or answer
   its own decision request.
4. **Report is state-surface edit + discrete event, governed by half-life.** A
   raw transcript is a firehose that destroys signal density and makes the next
   agent pay to re-read it (the cost `terse`/`freelunch` cut). Live progress
   edits one mutable surface in place; only *settling points* become durable git
   objects. This is `doctrine`'s lifetime rule applied to reports.
5. **git is the source of truth; the webhook stream is the trigger surface.**
   git is a poor real-time queue (no global order, branch-level eventual
   consistency, merge conflicts). Subscribe to events; do not poll git. Merge
   discipline is the serialization point — which is the oracle act anyway.
6. **Compose, don't duplicate.** `warrant` owns *what may begin*; `doctrine`
   owns *where documents land*; `freelunch` owns *execution shape*; `terse` owns
   *prose*. `dispatch` owns only the *channel*: how beginning, progress, and end
   are communicated and triggered.
7. **Waiting is a remote delivery, not a local hold.** Because the channel is
   git, a run ends by writing to the *remote* — a local write, or a session
   idling in-process, delivers nothing: no remote object means no event, so
   neither the upper human (notification) nor an upper agent (runner) is
   triggered, and the block is invisible. "Parked on the oracle" is therefore a
   property of *remote git state* (a labelled PR carrying an open decision
   marker), never of a running process. The agent does not wait; it delivers the
   blocking state to the remote and terminates. A later resolution is a fresh run
   that rebuilds state from the remote — nothing survives in a live local
   process, and nothing needs to.

## What will be done

A new steering plugin, `dispatch`, structured exactly like `warrant` (a
`UserPromptSubmit` directive for the judgment no path check can make, plus
`PreToolUse` gates for the mechanical halves, plus a `SessionStart` rebuild from
git). It ships three mechanisms.

### Mechanism A — the report ledger (outbound is a git write, by half-life)

The directive routes every report to a git object chosen by lifetime, mirroring
doctrine's buckets:

| Report | git object | half-life |
|---|---|---|
| live status | one mutable surface, **edited in place** — PR body checklist, or `.dispatch/status.md` on the branch | short — never becomes history |
| increment account | commit message + `Dispatch: <unit>` trailer | medium |
| finding / question / outcome | discrete git object at a **settling point** — PR/issue comment, decision doc, merge, close(state_reason) | long |

The rule the directive enforces as judgment: *promote to a discrete git event
only at a settling point; everything else edits the live surface.* Progress
narration never becomes a new comment. `git log --grep 'Dispatch: <unit>'` then
answers "what shipped for this unit" with no index maintained — the same trick
warrant uses.

### Mechanism B — the decision request (a blocking git state for "parked on oracle")

When the agent needs the human oracle mid-flow, a chat modal (`AskUserQuestion`)
is a **violation** — nothing can subscribe to it. Instead the agent writes a
blocking git object and stops:

- a comment carrying a machine-parseable marker —
  ```
  <dispatch:decision unit="<unit>" blocking="true">
  QUESTION:  <one line>
  OPTIONS:   A | B
  BLOCKED:   no default — this waits on the oracle
  </dispatch:decision>
  ```
- plus a `dispatch:awaiting-oracle` label on the PR (the visible, subscribable
  "parked" state).

The blocking state is **delivered to the remote in two coupled forms**, then the
run **terminates** (it does not idle in-process — constraint 7):

- a committed-and-pushed marker file `.dispatch/<unit>.decision.md` on the
  branch — the *git-native* truth, so a plain clone or a fresh `state.sh` can
  reconstruct "this unit is parked" after a `git fetch`, with no API call and no
  live session;
- the PR label + marker comment — the *subscribable trigger surface* the human
  and the runner watch.

A run is not "parked" until that push/post **succeeds**; until then the block is
invisible and the agent has not actually asked anyone. The human's resolution is
the remote git event that unblocks and re-triggers a fresh run (via the existing
`subscribe_pr_activity` path). `state.sh` does a `git fetch` and reads the
committed `.dispatch/*.decision.md` markers at session start, reporting which
units are parked on the oracle — symmetric to warrant's `state.sh`, but reading
the remote-delivered state, not a live process. So an interrupted decision
survives session death and a stranger can see exactly what is waiting on whom.

### Mechanism C — the transport rule (`report == trigger`)

The directive states plainly: for triggering and reporting, the only channel is
git; author every report so its git write is also the next actor's trigger. A
chat UI is allowed **only as a view over git-comment events** (a typed message
lands as a comment event), never as a side channel agents cannot subscribe to.

**The terminal invariant.** Every triggered run ends by leaving its whole
outcome as a *remote* git mutation and nothing in chat — the four shapes are:
produced (push commits / update PR), parked (deliver the blocking marker per
Mechanism B, terminate), in progress (edit the one live-status surface, push
WIP), dead-end (push a diagnosis comment + close/label). "Always push the result"
is only the *produced* case; the invariant is the wider **"terminate on the
remote,"** which includes writing a not-done, waiting state. This is precisely
what lets the runner be stateless and purely git-event-driven: because every run
terminates on the remote, the runner rebuilds all state from git each firing and
needs no memory between triggers. Part of the design (the runner is buildable)
and this invariant (runs terminate on the remote) are the same requirement from
two sides — a run that ended in chat could not be git-event-driven.

### Hooks (mirrors warrant's shape)

- `UserPromptSubmit` → `directive.sh`: injects `<dispatch-directive>` — surface
  gate, the ledger half-life rules, the decision-request convention, the
  `report == trigger` rule, the **remote-delivery / no in-process idle** rule
  (deliver the blocking or final state to the remote and terminate; never yield
  and wait locally; not "parked" until the push/post succeeds), composition, and
  a NEVER list. Judgment only.
- `PreToolUse` (`.*`) → `report-gate.sh`: input-only, mechanical, fails open —
  - refuses a `git commit` without the `Dispatch: <unit>` trailer while a unit
    is in flight (warrant's trailer check, for this unit id);
  - declines to vouch (falls back to the normal permission prompt) for writes
    that route around the bus, e.g. file writes to the live-status surface via
    shell redirection.
- `PreToolUse` (`.*`) → `decision-lock.sh`: input-only — refuses opening a
  *second* decision request while one is open (warrant's "never a second
  hunter" pattern), and refuses an action that would resolve / merge / answer
  the agent's *own* `awaiting-oracle` marker (constraint 3, enforced at the
  gate).
- `SessionStart` → `state.sh`: does a `git fetch` and reads the committed
  `.dispatch/*.decision.md` markers + branch state from git — reports which units
  are live and which are parked on the oracle from the *remote-delivered* state
  (not a live process), writes nothing.
- `SubagentStop` → releases the one-decision-in-flight lock (mirrors warrant's
  `hunt-state.sh release`).

All hooks carry the `DISPATCH_OFF=1` kill switch with the repo's `case` idiom and
fail open on missing `python3` / unreadable payload / unknown schema.

### Composition (explicit, so the gates never fight)

- **warrant**: approval of a warrant proposal is itself an oracle event; under
  dispatch it is a git act (an "approved" comment / label). The `Proposal:` and
  `Dispatch:` trailers name the same unit. warrant decides *what begins*;
  dispatch decides *how it is reported and what re-triggers it*.
  - *Worked case — the warrant hunter.* The hunter is the cleanest instance of
    the split dispatch closes: its **record** already lands as a git file
    (`docs/reports/<date>-hunt-<slug>.md`, a section left even on a null result),
    but its **finding** currently surfaces over chat ("reaches the user at the
    next turn boundary, in one line"; a before-landing hunter "reports into the
    landing exchange"). dispatch relocates that surfacing to the remote: a
    finding becomes a report comment on the unit's PR, or — when it must block
    the merge — a decision request with `awaiting-oracle` that parks landing on
    the oracle, feeding the merge decision exactly where warrant wants it. The
    hunter is a background *worker* (generation layer), so its output terminates
    on the remote too (constraint 7): the hunter, or the parent relaying it,
    pushes the record and posts the finding — the finding never lives only in the
    parent's chat context. Because both hunter moments are now git events
    (after-proposal = draft PR opened, before-landing = merge requested), the
    hunter dispatch is itself git-triggerable under the runner; within dispatch
    the invariant is only that the hunter's output is remote-delivered like any
    other report.
- **doctrine**: dispatch's settling-event reports that are *documents* (decision
  docs, reports) are placed by doctrine's gate. dispatch governs reports as
  *events*; doctrine governs documents as *files*. dispatch never places a file.
- **freelunch**: `report == trigger` is exactly what lets a fan-out compose
  without an orchestrator — each worker's report is a git write that triggers the
  next actor. dispatch is how freelunch's workers report and hand off.
- **terse**: terse compresses report *prose*; dispatch chooses the report
  *channel and granularity*. Orthogonal, no overlap.

## Out of scope

- No chat-UI implementation and no webhook infrastructure — the plugin steers an
  agent already running in a git-event environment (this session's
  `subscribe_pr_activity` is that environment). dispatch does not build the bus;
  it makes the agent speak only through it.
- No content audit or post-hoc report review — placement and granularity happen
  at write time, consistent with the stack's "no verification" thesis.
- No change to warrant/doctrine/freelunch/terse; composition is by convention
  (shared unit id, deferred file placement), not by editing them.
- Benchmarking. Like warrant and doctrine at v0.1.0, dispatch ships
  unbenchmarked; an ablation (does git-only reporting change wall-clock or
  hand-off correctness under fan-out?) is a later report.

## How we will know it worked

- With dispatch on, an agent that needs a decision pushes a labelled,
  marker-carrying git object **to the remote** and terminates the run — never an
  `AskUserQuestion` modal, never an in-process wait.
- The parked state is visible from a **different clone**: a fresh session on
  another machine runs `state.sh`, fetches, and names that unit as parked on the
  oracle — proving the block was delivered to the remote, not held locally, and
  that it survives session death.
- A commit made while a unit is in flight without the `Dispatch:` trailer is
  refused; with it, allowed.
- A second decision request while one is open is refused; an agent action that
  would resolve its own `awaiting-oracle` marker is refused.
- Progress across a multi-round task appears as in-place edits to one surface
  plus a small number of settling-point objects — not a comment per round.
- warrant and doctrine gates raise no new refusals on a normal dispatch flow
  (composition holds).
